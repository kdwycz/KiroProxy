## Context

KiroProxy 的 429 处理存在三个问题：

1. **冷却失效**：`mark_quota_exceeded()` 受 `rate_limiter.enabled`（默认 `False`）控制，429 后不创建冷却记录，账号始终被视为"可用"
2. **无效重试**：两个限速账号来回切换直到 `max_retries` 耗尽
3. **统计碎片化**：三套独立统计系统（`Account.request_count`、`StatsManager`、`FlowMonitor`），429 相关状态无法在 WebUI 展示

日志分析显示：89 次 429 中，80%+ 在 1~3 秒内恢复（瞬时限速），少数需要更长恢复期。

## Goals / Non-Goals

**Goals:**
- 429 冷却机制始终生效，不受限速开关控制
- 两阶段重试：快速重试 → 冷却+换号
- 指数退避冷却时间（5→10→30→60→120→300s）
- Web 后台展示账号限速状态
- 同一请求内不重复尝试已失败的账号

**Non-Goals:**
- 不改造统计系统整合（P1，后续做）
- 不改变 `rate_limiter.enabled` 对主动限速（请求间隔/RPM）的控制
- 不增加 `getUsageLimits` API 的自动查询（429 主要是瞬时限速）

## Decisions

### D1: 429 冷却独立于主动限速

**决定**：`mark_quota_exceeded()` 始终调用 `quota_manager.mark_exceeded()`，不再检查 `rate_limiter.enabled`。

**替代方案**：默认启用 `rate_limiter.enabled` → 会同时启用所有限速功能（请求间隔、RPM 限制），影响面太大。

### D2: 两阶段重试策略

**决定**：
- **第一阶段**：sleep(2~3s) 后用同一账号重试（覆盖大多数瞬时限速）
- **第二阶段**：触发指数退避冷却，执行换号策略

**理由**：日志中 80%+ 的 429 在 1~3 秒内恢复，第一阶段避免了不必要的冷却标记和账号切换。

### D3: 指数退避参数

**决定**：`5→10→30→60→120→300s`，每个账号独立计数，成功请求重置为第 1 级。

**替代方案**：

| 方案 | 序列 | 评价 |
|------|------|------|
| 3 倍递增 | 5→15→45→135→300 | 递增太快，第 3 级就 45s |
| 2 倍递增 | 5→10→20→40→80→160→300 | 级数太多 |
| **选定** | **5→10→30→60→120→300** | **节奏适中，6 级覆盖** |

### D4: 退避状态存储位置

**决定**：在 `QuotaManager` 的 `QuotaRecord` 中增加 `backoff_level` 字段，不持久化。

**理由**：退避级别是运行时状态，重启后从 0 开始是合理的。持久化退避级别反而可能导致重启后账号长期不可用。

### D5: 换号策略（策略 C 变体）

**决定**：

```
429 第一次 → sleep(2~3s) → 重试同账号
429 第二次 → mark_cooldown → 检查其他可用账号
  ├─ 有可用账号 → 切换
  └─ 无可用账号 → 等最短冷却剩余时间（≤5s 时 sleep 等待）
```

`get_next_available_account()` 支持传入 `exclude_ids: Set[str]` 排除已试账号。

## Risks / Trade-offs

- **[快速重试可能加重限速]** → 第一阶段的 sleep(2~3s) 已提供了足够缓冲，且日志验证了 1~3s 恢复的普遍性
- **[退避级别不持久化]** → 重启后归零，可能在重启后短暂触发密集 429。但因为退避从 5s 起步，影响很小
- **[所有 handler 需改动]** → 三个 handler（Anthropic/OpenAI/Gemini）需同步改动重试循环。可将重试逻辑提取为共享函数降低重复
